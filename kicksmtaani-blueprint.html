Section 3
System Architecture
How all the pieces connect — from customer browser to payment confirmation.
🌐 Request Flow — Customer Places an Order
1

Customer visits kicksmtaani.co.ke

Browser hits Vercel Edge CDN → Next.js serves SSR page with product data. Page load <2s even on 3G.

2

Browse + Add to Cart

Cart state managed in Zustand (localStorage-persisted). User selects size, category (Men/Women/Kids), quantity.

3

Checkout — Login or Guest

JWT auth flow. Guest checkout allowed (email + phone required). Session stored in Redis for cart recovery.

4

Payment Selection

M-Pesa STK Push (phone number → Safaricom prompt) OR Card via Flutterwave. Flutterwave also supports M-Pesa.

5

Order Created (Atomic Transaction)

PostgreSQL transaction: decrement inventory → create order → create payment_intent. Either all succeed or all roll back.

6

Payment Webhook Received

Safaricom/Flutterwave POSTs to /api/webhooks/mpesa. Webhook verified with HMAC signature. Order status → CONFIRMED.

7

Notifications Fired

BullMQ job queue sends: SMS confirmation to customer, WhatsApp message, email receipt. Admin gets Slack notification.

Refresh Token:
  - 32-byte random hex

⚡ Key Database Rules
  • ✓ All monetary values stored as DECIMAL(12,2) — never FLOAT (floating point rounding kills money calculations)
  • ✓ Stock decrements use SELECT FOR UPDATE to prevent race conditions on concurrent purchases
  • ✓ Soft deletes on products (is_active=false) — never DELETE rows that have order history
  • ✓ All timestamps in UTC. Display conversion happens in the frontend.
  • ✓ UUID primary keys — prevents enumeration attacks, scales to multiple DB shards
  • ✓ Indexes on: user_id, category, status, created_at, slug — all common query columns

Section 7
Security Architecture
OWASP Top 10 addressed. Every attack vector defended. Security is not an afterthought.
🛡️ API Security
🔐

JWT + HttpOnly Cookies

Access tokens in memory only. Refresh tokens in httpOnly, Secure, SameSite=Strict cookies. XSS cannot steal tokens.

🚦

Rate Limiting

express-rate-limit per IP. Auth routes: 5 req/15min. General: 100 req/15min. Redis-backed for multi-instance accuracy.

🧹

Input Validation

Every request body validated with Zod. SQL injection impossible with Prisma parameterized queries. No raw SQL.

🪖

Helmet.js

Sets 15+ security HTTP headers: CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy.

🌐

CORS

Strict allowlist: only kicksmtaani.co.ke and admin.kicksmtaani.co.ke. No wildcard origins in production.

🔒 Data Security
🔑

Password Hashing

bcrypt with 12 rounds. Never store plain text. Never log passwords. Never return password_hash in API responses.

🔏

Secrets Management

All secrets in environment variables. Never in code or git. Use Railway/Vercel secret injection. Rotate on breach.

📊

Audit Logging

All admin actions logged: who, what, when, IP. Immutable audit_log table. Helps detect insider threats.

💾

Database Security

API uses least-privilege DB user (no DROP/CREATE). DB not publicly accessible. SSL-only connections. Regular backups.

🪝

Webhook Verification

Every M-Pesa/Flutterwave callback verified with HMAC-SHA256. Prevents fake payment confirmations.

⚙️ Security Middleware Order (Express)
// Apply in this EXACT order — order matters
app.use(helmet())                    // 1. Security headers FIRST
app.use(cors(corsOptions))           // 2. CORS allowlist
app.use(express.json({ limit: '10kb' }))  // 3. Body limit (prevent huge payload attacks)
app.use(globalRateLimit)             // 4. General rate limiting

📲 M-Pesa STK Push Flow (Step by Step)
1

Customer clicks "Pay with M-Pesa"

Enters phone number (07XX or 01XX). Frontend sends to POST /api/v1/payments/mpesa/stk-push

2

Server gets OAuth token from Safaricom

Calls Daraja /oauth/v1/generate?grant_type=client_credentials with Base64(ConsumerKey:ConsumerSecret). Token cached in Redis for 55 min.

3

Server sends STK Push request

POST to mpesa.safaricom.co.ke/mpesa/stkpush/v1/processrequest with BusinessShortCode, Timestamp, Password (SHA256 hash), Amount, PartyB, CallBackURL.

4

Customer sees popup on their phone

"KicksMtaani is requesting KES 2,500 from your M-Pesa account." Customer enters M-Pesa PIN.

5

Safaricom POSTs callback to our server

Hits /api/v1/webhooks/mpesa. Server verifies callback, extracts ResultCode, MpesaReceiptNumber, Amount, TransactionDate.

6

Order confirmed if ResultCode === 0

PostgreSQL transaction: payment status → SUCCESS, order status → CONFIRMED. SMS + email fired via BullMQ queue.

7

Polling fallback (if callback delayed)

Frontend polls GET /api/v1/orders/:id/payment-status every 3 seconds for up to 90 seconds. Shows real-time status to customer.

💻 M-Pesa STK Push Code Example
// services/mpesa.service.ts
export async function initiateSTKPush(params: {
  phone: string, amount: number, orderId: string
}) {
  const token = await getMpesaToken();
  const timestamp = getMpesaTimestamp(); // YYYYMMDDHHmmss
  const password = Buffer.from(
    `${process.env.MPESA_SHORTCODE}${process.env.MPESA_PASSKEY}${timestamp}`
  ).toString('base64');

  const res = await fetch(MPESA_STK_URL, {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}` },
    body: JSON.stringify({
      BusinessShortCode: process.env.MPESA_SHORTCODE,
      Password: password, Timestamp: timestamp,
      TransactionType: 'CustomerPayBillOnline',
      Amount: Math.ceil(amount),  // M-Pesa needs integer
      PartyA: formatPhone(params.phone), // 2547XXXXXXXX
      PartyB: process.env.MPESA_SHORTCODE,
      PhoneNumber: formatPhone(params.phone),
      CallBackURL: `${process.env.API_URL}/api/v1/webhooks/mpesa`,
      AccountReference: `KM-${params.orderId.slice(0,8)}`,
      TransactionDesc: 'KicksMtaani Order'
    })
  });

  if (!res.ok) throw new PaymentError('STK push failed');
  return res.json();
}

🧪 Testing Strategy
  • ✓ Unit tests (Jest) — Business logic, validators, utils. Target 80%+ coverage.
  • ✓ Integration tests — API endpoints hit real test DB. Order flow, auth flow, payment webhooks.
  • ✓ E2E tests (Playwright) — Full browser: browse → cart → checkout → confirmation.
  • ✓ Load tests (k6) — 100 virtual users, checkout scenario. Catches N+1 queries.
  • ✓ Test naming — describe('OrderService') → it('should decrement stock atomically')
🏛️ Code Architecture Principles
  • ✓ Repository pattern — DB queries in /repositories, not scattered in controllers
  • ✓ Service layer — Business logic in /services. Controllers just handle HTTP, delegate to services.
  • ✓ DTO validation — Zod schemas as DTOs. Validate at the boundary (controller), trust inside.
  • ✓ Error hierarchy — Custom error classes: AppError → ValidationError → AuthError → NotFoundError
  • ✓ No magic numbers — Constants in /constants/index.ts: ORDER_STATUS, ROLES, LIMITS
  • ✓ Async/await everywhere — No callback hell. Unhandled promise rejections treated as errors.
  • ✓ Single responsibility — Each function does ONE thing. Max 50 lines per function.
  • ✓ Semantic commits — feat: add size filter / fix: prevent negative stock / chore: update deps
